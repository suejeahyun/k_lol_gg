# K-LOL.GG 운영·보안·백업·복구 런북

이 문서는 운영 배포를 안전하게 반복하고, 장애나 데이터 손실이 발생했을 때 복구하기 위한 기준 절차다. 실제 비밀번호, 토큰, 데이터베이스 URL은 이 문서나 저장소에 기록하지 않는다.

## 1. 운영 책임과 접근

- Vercel 프로젝트와 데이터베이스에는 운영에 필요한 최소 인원만 접근시킨다.
- 슈퍼 어드민은 일반 관리자와 분리된 전용 계정을 사용하고 TOTP 2단계 인증을 등록한다.
- `SUPER_ADMIN_ID`, `SUPER_ADMIN_PASSWORD`, `JWT_SECRET`, `ADMIN_TOKEN_VALUE`, `CRON_SECRET`, `TOTP_ENCRYPTION_KEY`와 Kakao/Riot 비밀값은 Vercel Environment Variables에서만 관리한다.
- `TOTP_ENCRYPTION_KEY`는 최소 32자의 강한 랜덤 값으로 설정하고 별도의 암호화된 비밀 관리소에도 복구 사본을 보관한다. 이 키를 잃으면 저장된 관리자 TOTP 비밀키를 복호화할 수 없다.
- 비밀값을 채팅, 이슈, 로그 또는 스크린샷에 남기지 않는다. 노출이 의심되면 즉시 폐기하고 새 값으로 교체한다.
- 환경변수를 바꾼 뒤에는 새 프로덕션 배포를 만들고 기존 로그인 세션과 API 호출을 다시 확인한다.
- 공개 개인정보 문의처를 `PRIVACY_CONTACT`에 설정하고 실제로 응답 가능한 채널인지 분기마다 확인한다.

## 2. 배포 전 체크리스트

1. 변경 사항과 적용 대상(Preview/Production)을 확인한다.
2. 데이터 모델 변경이 있으면 `prisma/migrations`에 새 마이그레이션이 포함됐는지 확인한다. 이미 적용된 마이그레이션 파일은 수정하거나 삭제하지 않는다.
3. 배포 직전 복구 가능한 데이터베이스 스냅샷 또는 덤프가 있는지 확인한다.
4. 깨끗한 의존성 설치 후 아래 검사를 통과시킨다.

```bash
npm ci
npm run check:release
npm run migrate:status
```

5. 데이터베이스 스키마 변경은 CI/CD의 프로덕션 단계에서 `npm run migrate:deploy`로 적용한다. 개발용 `prisma migrate dev`나 `prisma db push`를 프로덕션 데이터베이스에 사용하지 않는다.
6. 마이그레이션과 애플리케이션 배포 순서는 이전 버전과 새 버전이 잠시 함께 실행돼도 안전하도록 설계한다. 파괴적 컬럼 삭제는 추가-전환-삭제의 여러 배포로 나눈다.

## 3. 배포 직후 검증

아래 순서로 확인하고 실패하면 신규 트래픽을 늘리지 않는다.

1. `https://k-lol-gg.vercel.app/api/health`가 HTTP 200과 `status: "ready"`를 반환하는지 확인한다.
2. 운영 주소를 `NEXT_PUBLIC_BASE_URL`에 설정한 환경에서 `npm run check:deploy-api`를 실행한다.
3. 데스크톱과 모바일에서 홈, 플레이어, 경기, 랭킹, 구인, 로그인 화면을 직접 확인한다.
4. Vercel Runtime Logs에서 신규 5xx, Prisma 연결 오류, 인증 오류 급증이 없는지 확인한다.
5. 관리자에서 최근 오류 로그, Riot 실패 로그, 필수 환경변수 경고를 확인한다.

`/api/health`는 데이터베이스 연결까지 점검하며 실패하면 503을 반환한다. 응답에는 비밀값, 데이터베이스 주소, 오류 상세를 포함하지 않는다.

## 4. 관측과 알림

- Vercel Observability에서 함수 5xx 비율, 응답 지연, 트래픽 및 사용량 급증을 매일 확인한다.
- 배포 후 최소 30분 동안 오류율과 `/api/health`를 집중 관찰한다.
- 외부 업타임 모니터가 있다면 `/api/health`를 5분 간격으로 확인하고 2~3회 연속 실패 시 운영자에게 알리도록 한다.
- 경고 기준 예시: 5xx 급증, health 503, DB 연결 오류, 로그인 실패 급증, Riot/Kakao 호출 실패 급증, 함수 실행 시간 급증.
- 로그에는 사용자 비밀번호, 토큰, 전체 요청 본문, 데이터베이스 URL을 남기지 않는다.

## 5. 백업 정책

- 데이터베이스 제공자의 자동 스냅샷이 활성화됐는지 매월 확인한다. 보존 기간과 가장 오래된 복구 지점을 기록한다.
- 자동 스냅샷과 별도로 중요한 변경 전에는 수동 스냅샷 또는 `pg_dump`를 만든다.
- 관리자 CSV 내보내기는 운영 확인과 부분 데이터 복구를 위한 보조 수단이다. 관계, 인덱스, 계정 상태 전체를 보존하는 데이터베이스 백업을 대체하지 않는다.
- 백업 파일은 원본 데이터베이스와 다른 보안 경계에 암호화해 보관하고 접근 권한을 최소화한다.
- 백업 파일명에는 UTC 생성 시각과 대상 환경을 포함하며, 성공 여부와 보존 만료일을 운영 기록에 남긴다.
- 백업 생성만으로 완료 처리하지 않는다. 최소 분기 1회 격리된 테스트 데이터베이스로 복구 훈련을 수행한다.

## 6. 복구 절차

1. 쓰기 작업을 중단하거나 점검 공지를 활성화해 추가 데이터 변경을 막는다.
2. 장애 시작 시각, 영향 범위, 마지막 정상 배포와 마지막 정상 백업을 기록한다.
3. 복구 대상 백업의 생성 시각과 무결성을 확인한다.
4. 가능하면 새 격리 데이터베이스에 먼저 복원한다. 기존 프로덕션 DB를 즉시 덮어쓰지 않는다.
5. `npm run migrate:status`로 마이그레이션 상태를 확인하고 필요한 경우 `npm run migrate:deploy`를 실행한다.
6. 격리 환경에서 관리자 로그인, 플레이어/경기 수, 최근 데이터, 핵심 관계와 `/api/health`를 검증한다.
7. 검증된 데이터베이스로 연결을 전환한 뒤 새 배포를 만들고 배포 직후 검증 절차 전체를 수행한다.
8. 유실 가능 시간 범위와 사용자 영향을 기록하고, 필요한 경우 운영 공지를 게시한다.

## 7. 장애 대응과 롤백

- 코드 장애이고 DB 스키마가 이전 버전과 호환되면 Vercel의 직전 정상 배포로 롤백한다.
- DB 변경이 포함된 경우 애플리케이션만 먼저 롤백하지 말고 스키마 호환성을 확인한다.
- 인증 또는 비밀값 유출이 의심되면 관련 키를 모두 회전하고 관리자 세션을 무효화한다.
- `TOTP_ENCRYPTION_KEY`는 일반 비밀값처럼 즉시 값만 바꾸면 기존 2FA가 모두 복호화 불가능해진다. 교체 전 기존 키로 값을 복호화해 새 키로 재암호화하는 마이그레이션을 수행하거나, 모든 관리자 2FA를 초기화하고 재등록하는 점검 절차를 먼저 준비한다.
- 장애 종료 조건은 health 정상, 핵심 스모크 테스트 통과, 5xx 안정화, 데이터 무결성 확인이다.
- 종료 후 원인, 탐지 시각, 대응, 영향, 재발 방지 작업과 담당자를 남긴다.

## 8. 정기 운영 일정

- 매일: 오류 로그, health, 백업 성공 여부, Riot/Kakao 실패 확인.
- 매일: Vercel Cron이 UTC 18시(한국 시간 다음 날 03시대)에 `GET /api/cron/maintenance`를 호출해 감사 로그와 요청 제한 로그를 정리한다. `CRON_SECRET`은 16자 이상의 랜덤 값으로 설정한다.
- 감사 로그는 `ADMIN_LOG_RETENTION_DAYS`(기본 365일, 30~3650일), 요청 제한 로그는 `RATE_LIMIT_LOG_RETENTION_DAYS`(기본 7일, 1~90일)를 따른다.
- 매주: `npm run check:deploy-api`, 관리자 권한 변경과 보안 로그 검토.
- 매월: 의존성 보안 업데이트 검토, 비밀값과 접근자 목록 점검, 백업 보존 확인.
- 분기: 격리 환경 복구 훈련, 슈퍼 어드민/TOTP 복구 절차 시험.

Cron 정리는 오래된 행을 조건 삭제하는 멱등 작업이므로 중복 호출에도 결과가 누적되지 않는다. Vercel은 실패한 Cron을 자동 재시도하지 않으므로 Cron Jobs의 실행 로그와 5xx를 매일 확인하고, 실패 알림은 별도 업타임/관측 도구에서 연결한다.
